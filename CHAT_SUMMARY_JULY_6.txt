## Payment Status Explained

| Status | Meaning |
|--------|---------|
| `created` | Order exists on Razorpay, user hasn't completed payment |
| `paid` | Payment confirmed via webhook |
| `failed` | User's payment was rejected (card declined, etc.) |
| `refunded` | Money returned |

- "Created" = user opened Razorpay but closed without paying OR payment still processing
- Proxy retries (5 times) only affect ORDER CREATION, not the payment itself
- If payment fails on Razorpay side, user must try again manually

## Wallet Balance
- Shows ₹0.00 because no payments are "paid" yet
- Once a payment is confirmed via webhook, ledger updates and balance increases
- Razorpay settles to bank in T+2 days

## Multi-App Payment Architecture

### Current Setup (supports 2-3 apps)
```
App 1: sk_live_abc123 → Razorpay key_id: rzp_test_xxx
App 2: sk_live_def456 → Razorpay key_id: rzp_test_xxx (same)
App 3: sk_live_ghi789 → Razorpay key_id: rzp_test_xxx (same)
```

- Different API keys per app (can revoke independently)
- Same Razorpay account (one dashboard, one bank settlement)
- Per-app tracking via `appId` in transactions

### ProviderKey entries (same Razorpay creds for all apps)
```javascript
{ appId: app1_id, provider: "razorpay", keyId: "rzp_test_xxx", keySecret: "secret_xxx" }
{ appId: app2_id, provider: "razorpay", keyId: "rzp_test_xxx", keySecret: "secret_xxx" }
{ appId: app3_id, provider: "razorpay", keyId: "rzp_test_xxx", keySecret: "secret_xxx" }
```

## Multi-Profile System (Planned)

### Production-Ready Architecture
```
Browser:
  HTTP-only cookie: session_id=abc123    (secure, can't read via JS)
  localStorage: { activeAppId: "def" }   (just UI state)

Server (Redis):
  session_id: abc123
  └─ userId: "user123"
     activeAppId: "def"
     tokens: {
       "app_abc": "jwt_xxx",
       "app_def": "jwt_yyy"
     }
```

### Why This Approach
- HTTP-only cookie = XSS protection
- SameSite=Strict + CSRF token = CSRF protection
- Redis stores all tokens, cookie only holds session ID
- Server-side token refresh, user never sees tokens
- Same pattern used by Vercel, Stripe Dashboard, Linear

### UI Design
```
Sidebar bottom:
  ┌──────────────────────┐
  │ 🟢 Store      ▼      │  ← active app
  │ ──────────────────── │
  │ 🟢 SaaS              │  ← other apps
  │ ──────────────────── │
  │ + Add new app        │  ← opens login
  └──────────────────────┘
```

### Redux State
```
apps: []           // all saved app sessions
activeAppId: ""    // which one is active
```

### Actions
- `switchApp(appId)` → sets activeAppId, refetches dashboard
- `addApp(appData)` → pushes to apps array, sets as active
- `removeApp(appId)` → removes from array

### What to Add
- `express-session` + `connect-redis` for session middleware
- `sessions` collection in MongoDB (or Redis) keyed by userId
- `/switch-app` endpoint that updates the cookie
